5. Implementation

This chapter describes how the Attendance System was implemented after completing the requirement analysis and system design phases. It explains how the business requirements were transformed into actual software components, including environment preparation, application development, testing, deployment procedures, and production operations. The implementation process was divided into multiple environments to ensure that new features could be developed, tested, validated, and deployed safely.

5.1 Environment Setup

To support the development lifecycle, three main environments were used throughout the project: Local Development and Testing, Pre-Production (Staging), and Production. Each environment served a different purpose and helped ensure software quality before deployment to end users.

5.1.1 Local Development and Testing Environment

The first stage of development was performed on local machines. A Docker-based environment was created to provide a consistent and isolated workspace for development and testing. Multiple containers were used to run the required services, including the PostgreSQL database, Django backend API, Telegram bot, and Mini App web server.

For the Mini App, Nginx was used to host the static HTML, CSS, and JavaScript files. This allowed the Telegram Mini App to be tested locally before deployment.

Since the system relied heavily on Telegram integration, a Telegram Bot was created using BotFather. In addition, separate Telegram groups were configured for employees and HR staff to support attendance notifications and operational testing.

To simulate production conditions, the Faker Python library was used to generate large volumes of test data. Employee records, attendance logs, departments, and other related data were generated to resemble real-world usage scenarios. The database structure in the testing environment was kept consistent with the production database schema to ensure accurate testing results.

Telegram Mini Apps require HTTPS connections; therefore, the local environment needed a secure public URL. VS Code Tunneling was used to expose local services through HTTPS. Compared to alternatives such as Ngrok or Cloudflare Tunnel, VS Code Tunneling provided a stable development URL and reduced the need for frequent configuration changes. It also enabled supervisors and team members to access and test the application remotely during development.

This environment provided a fast and flexible platform for implementing new features, debugging issues, and performing initial validation before moving changes to the staging environment.

5.1.2 Pre-Production (Staging) Environment

After successful local testing, new features were deployed to the Pre-Production environment for integration testing and user acceptance validation. This environment was hosted on an internal Debian Linux server known as Docker01.

The Docker architecture was similar to the local environment but was configured to more closely resemble the production infrastructure. Multiple containers were deployed for the backend application, Mini App, cron jobs, and supporting services.

To improve security and simplify network management, Nginx was configured as a reverse proxy. External traffic was exposed through a single entry point while requests were internally routed to the appropriate services. Requests to /api were forwarded to the backend API service, while requests to /miniapp were routed to the Mini App service.

The staging environment served as the primary platform for validating new releases using real business scenarios. Production data was copied into the staging environment using PostgreSQL backup and restoration tools such as pg_dump, allowing testing to be performed with realistic datasets and organizational structures.

Deployment procedures were also tested extensively during this stage. Activities included:

Verifying deployment branches.
Pulling the latest source code from the repository.
Rebuilding and restarting Docker containers.
Executing database migrations when schema changes were required.
Verifying application functionality after deployment.

Rollback procedures were also tested to ensure system recovery in case deployment failures occurred. These simulations helped establish a reliable deployment process and reduced operational risks when releasing updates to production.

5.1.3 Database Infrastructure

The Attendance System uses PostgreSQL as its primary database management system. Instead of hosting separate databases on each application server, the company maintains a dedicated shared database server known internally as MonDB.

Both the Pre-Production and Production environments connect to this centralized PostgreSQL server. This architecture simplifies database administration, backup management, and monitoring while allowing multiple internal applications to share the same database infrastructure.

To support testing activities, production database snapshots were periodically copied to the staging environment. This ensured that testing was performed using realistic data volumes and business conditions, allowing potential issues to be identified before production deployment.

5.1.4 Production Environment

The Production environment is the live system used daily by company employees. Due to organizational security policies, deployment activities were primarily performed by System Engineers, while my involvement was limited to assisting with deployment preparation, validation, and post-deployment testing.

The production architecture is similar to the staging environment but runs on dedicated infrastructure designed for operational use. Standard HTTP and HTTPS ports (80 and 443) are exposed externally, while Nginx routes incoming requests to the appropriate internal services.

Before each deployment, the release package was validated in the staging environment. Once approved, System Engineers performed the deployment process, including source code updates, container restarts, database migrations, and service verification.

The production environment also includes monitoring and logging mechanisms that record system activities and user interactions. These logs are essential for troubleshooting, auditing, and diagnosing operational issues. By analyzing application logs, administrators can quickly identify errors, investigate user-reported issues, and maintain system stability.

The use of separate development, staging, and production environments significantly improved software quality and reduced deployment risks throughout the project lifecycle.
